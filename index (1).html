NIVEL 1 // Cross-Site Scripting (XSS)

bloqueado

En la primera versión del kiosco, cuando cargabas un cliente al fiado, su nombre se insertaba directo en la página sin ningún filtro. Si alguien escribía código en vez de un nombre, ese código se ejecutaba en el navegador. En la app real esto se arregló con una función escapeHtml() que "neutraliza" cualquier código antes de mostrarlo.

SANDBOX // Alta de cliente (versión vulnerable, sin escapeHtml)
Objetivo: lograr que se ejecute código JavaScript a través del campo "Nombre del cliente", de forma que se llame a la función capturarFlag('L1'). Si lo lográs, va a aparecer la flag abajo sola.
💡 Pista
El campo se inserta con innerHTML, no con texto plano. Un tag de imagen con un atributo onerror se ejecuta apenas el navegador intenta cargarla y falla. Probá algo como:
<img src=x onerror="capturarFlag('L1')">
🛡 Cómo se arregla (la versión real)
En vez de innerHTML = nombre, se pasa el nombre por una función que reemplaza los caracteres peligrosos (< > " ' &) por su versión "de texto", así el navegador los muestra como letras y no como código. Es exactamente el escapeHtml() que le agregamos a los nombres de cliente, producto y detalle de fiado en la app real.

NIVEL 2 // Autorización solo del lado del cliente

bloqueado

Este panel decide quién es "administrador" mirando un dato guardado en tu propio navegador (localStorage). El problema: ese dato lo controlás vos mismo, no un servidor. Es el mismo tipo de chequeo que usa el sistema de licencias del kiosco real (bloquear un kiosco que no pagó) — por eso ahí mismo te explicamos que es una protección liviana, no a prueba de balas.

SANDBOX // Panel de administrador
🔒 ACCESO DENEGADO — no sos administrador
Objetivo: conseguir acceso de administrador sin que exista ningún botón de "login" en esta página.
💡 Pista
Abrí la consola del navegador (F12 → pestaña "Console") y escribí:
localStorage.setItem('l2_esAdmin', 'true')
El panel se actualiza solo, sin necesidad de recargar la página.
🛡 Cómo se arregla (el concepto real)
La regla de oro: nunca confiar en un dato que vive del lado del navegador para decidir permisos importantes. Un servidor real tiene que validar la identidad con algo que el usuario no pueda tocar (una sesión, un token firmado). En el kiosco real, por eso el chequeo de licencias solo bloquea con un cartel (una molestia, no una caja fuerte) — no maneja nada crítico como plata o datos de otros. Para algo realmente sensible (como el backend de Mercado Pago), la clave (APP_SHARED_SECRET) vive en el servidor, no en el navegador.

NIVEL 3 // Secretos expuestos en el código fuente

bloqueado

Todo lo que le llega al navegador — HTML, CSS, JavaScript — lo puede leer cualquiera, aunque no se muestre en pantalla. Guardar una clave secreta "escondida" en el código del frontend no la esconde realmente. Por eso el Access Token de Mercado Pago nunca vive en la app — vive únicamente en el backend, en variables de entorno que el navegador jamás llega a descargar.

SANDBOX // Esta página tiene un secreto escondido en su propio código

No hay ningún botón para este nivel. La vulnerabilidad ya está en el código de esta misma página, esperando a que la encuentres.

Objetivo: encontrar la flag mirando el código fuente de esta página.
💡 Pista
Click derecho → "Ver código fuente de la página" (o Ctrl+U). Buscá con Ctrl+F la palabra CONFIG_INTERNA.
🛡 Cómo se arregla (el concepto real)
Ninguna clave, contraseña o token va en código que se le manda al navegador — ni en el HTML, ni en el JS, ni siquiera "ofuscado" o en un comentario. Si algo necesita una clave secreta para funcionar (como llamar a la API de Mercado Pago), esa llamada la tiene que hacer el servidor, no el navegador del kiosquero.

NIVEL 4 // Referencia insegura a objetos (IDOR)

bloqueado

Esta mini-pantalla te muestra los datos de una cuenta según un ID que vos mismo escribís, sin verificar si esa cuenta es realmente tuya. Es el error clásico que evitamos en el sistema de licencias: cada kiosco solo debería poder ver o afectar su propia información, nunca la de otro, sin importar qué ID escriba.

SANDBOX // Ver cuenta por ID
Objetivo: tu cuenta es la 1. Conseguí ver los datos de una cuenta que no es la tuya.
💡 Pista
Probá cambiar el número del campo por otros valores cercanos, como 2 o 3.
🛡 Cómo se arregla (el concepto real)
Cada pedido de datos tiene que verificar del lado del servidor "¿esta persona tiene permiso de ver este ID en particular?" — no alcanza con que el ID exista. En un sistema con login real, eso se resuelve comparando el ID pedido contra el ID de la sesión autenticada del usuario, no confiando en lo que el navegador mande.

NIVEL 5 // Confiar ciegamente en avisos externos

bloqueado

Esta es una compra simulada de $5000. El botón "Comprar" debería ser el único camino para que la compra quede marcada como pagada. Esto es un espejo del webhook de Mercado Pago: antes de agregarle la validación de firma, cualquiera podía mandarle a tu backend un aviso falso de "ya se pagó" sin haber pagado nada.

SANDBOX // Compra simulada
Estado: 🔴 NO PAGADO
Objetivo: lograr que el estado pase a "PAGADO" sin tocar el botón "Comprar".
💡 Pista
El botón "Comprar" en realidad llama a una función interna. Esa misma función quedó accesible desde la consola del navegador, como si fuera un endpoint sin protección. Probá escribir en la consola: confirmarPagoL5()
🛡 Cómo se arregla (la versión real)
En el backend real, el webhook de Mercado Pago ahora valida una firma criptográfica (x-signature) antes de creerle a cualquier aviso de pago. Esa firma se calcula con una clave secreta que solo Mercado Pago y tu servidor conocen — nadie puede fabricar un aviso falso sin esa clave, a diferencia de esta versión de juguete, donde "confirmar el pago" es una función abierta que cualquiera puede llamar.
